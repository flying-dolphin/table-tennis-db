# V1 完整任务计划

日期：2026-04-17
关联文档：`docs/design/PRD-v1.md`、`docs/design/database.md`、`docs/design/backend.md`、`docs/design/frontend.md`

---

## 总体说明

本文档为 V1 的全部开发任务。共 8 个 Task，覆盖 PRD 中所有 V1 必做页面和功能。

### 依赖关系

```
Task 1（数据层）
  └─> Task 2（API 骨架）
        ├─> Task 3（首页）
        ├─> Task 4（排名页 + 对比页）
        ├─> Task 5（球员详情页）
        ├─> Task 7（赛事页 + 比赛页）
        └─> Task 6（账号体系）
              └─> Task 8（搜索结果页）
```

Task 1、2 为基础设施，必须先行。Task 3-7 可适度并行。Task 8 依赖 Task 6 的账号和权限体系。

---

### Task 1：后端数据层与 schema 对齐

目标：把现有临时 SQLite 结构切到新设计基线，至少保证后续 API 能站在统一数据模型上。

当前现状（见 `database.md` 第 0 章）：
- 实际 DB 仅 7 表，唯一有实质数据的是 `player_profiles`（101 条）
- 旧 `players` 表和 `events`/`matches` 表结构与目标完全不同
- 目标 schema 需要 14+ 表

范围：

- 对齐 `docs/design/database.md` 中的新赛事分类体系
- 重写 `scripts/db/schema.sql` 作为唯一正式 schema 源
- 删除 `web/scripts/sync-to-db.ts`，退出 web 侧临时入库链路
- 从 `player_profiles` 迁移数据到新 `players` 表
- 明确首页、排名页、球员页、赛事页、比赛页最小必需数据表与字段

产出：

- 新版 schema（覆盖 database.md 中的所有目标表）
- 可运行的迁移/初始化脚本
- 正式入库链路改为 `scripts/db/import*.py`，不再使用 web 侧临时同步脚本
- `import_points_rules.py` 保留到后续实现阶段，本轮不要求导入 `points_rules`

### Task 2：打通 /api/v1 基础公开接口

目标：建立新 API 骨架，逐步替代旧 `app/api/rankings` 和 `app/api/players/[slug]`。

优先接口：

- `GET /api/v1/home/calendar`
- `GET /api/v1/home/rankings`
- `GET /api/v1/rankings`
- `GET /api/v1/players/{slug}`
- `GET /api/v1/compare`

原因：

- 这 5 个接口能支撑首页、排名页、球员详情页和对比页
- 能最早把前端从 mock/JSON 读取迁到 API 驱动
- compare 接口为 Task 4 主链路所必需

产出：

- 统一响应格式（PRD 11.3）
- `lib/api` 或等价 server-side data access 层
- 旧 API 标记为待废弃

### Task 3：首页改造为真实数据驱动

目标：把当前首页原型升级为符合 PRD 的首页。

范围：

- SearchBox 改成"未登录可见、点击跳登录"
- EventScroller 去掉 mock，接 `home/calendar`
- RankingTable 去掉 mock，接 `home/rankings`
- 底部导航改成 首页 / 排名 / 赛事 / 用户

涉及文件：

- `web/components/home/SearchBox.tsx`
- `web/components/home/EventScroller.tsx`
- `web/components/home/RankingTable.tsx`
- `web/components/BottomNav.tsx`
- `web/app/page.tsx`

### Task 4：实现排名页与对比页主链路

目标：优先落地 PRD 里最核心的主链路：首页 -> 排名页 -> 选择两个球员 -> 对比页

范围：

- 新建 `/rankings` 页面
- 支持三种排序（积分 / 胜率 / 交手次数）
- 支持勾选两个球员
- 新建 `/compare` 页面
- 接 `GET /api/v1/rankings` 和 `GET /api/v1/compare`

这是第一批里最重要的功能任务，因为它最能验证产品差异化价值。

### Task 5：重做球员详情页，按 PRD 缩放到正式结构

目标：保留现有球员页路由，但重构内容结构和数据来源。

范围：

- 顶部信息区改为 PRD 定义
- 核心统计对齐新口径
- 最近比赛固定最近 3 场
- 比赛记录改为 events list
- 增加 Top 3 对手
- 移除旧的临时统计逻辑

涉及文件：

- `web/app/players/[slug]/page.tsx`
- 同时替换 `web/lib/data.ts` 旧逻辑

### Task 6：实现赛事列表页、赛事详情页与比赛详情页

目标：落地 PRD 第 8 章的赛事维度浏览链路。

范围：

- 新建赛事列表页 `/events`，支持按年份切换，回溯到 2014
- 新建赛事详情页 `/events/[eventId]`
  - 支持多 sub event 切换
  - 默认规则：普通赛事默认女单，团体赛默认女子团体
  - 展示正赛对战图和晋级路径
  - 展示各 sub event 冠军
  - 无数据的 sub event 置灰
- 新建比赛详情页 `/matches/[matchId]`
  - 展示赛事名称、sub event、轮次、日期、双方球员、国家、比分、局分明细、获胜方
  - 提供返回所属赛事入口

对应后端接口：

- `GET /api/v1/events?year=...`
- `GET /api/v1/events/{eventId}?sub_event=...`
- `GET /api/v1/matches/{matchId}`

跳转关系：

- 赛事列表页 -> 赛事详情页
- 赛事详情页对战图 -> 比赛详情页
- 比赛详情页 -> 所属赛事

### Task 7：实现最小账号体系和搜索拦截

目标：先把"登录后可搜"的门槛搭起来，再做复杂搜索。

范围：

- `/auth` 登录/注册页
- `/me` 用户页
- 注册 / 登录 / 登出 / me 接口
- 首页搜索框和全局搜索点击拦截
- 登录后回首页
- 验证码发送机制
- 基础限流（注册 / 登录接口）

原因：

- 搜索是 V1 重点，但完整 NLQ 搜索和 mixed_result 比较重
- 第一批先把账号和访问控制落地，能降低后续返工


### Task 8：实现搜索结果页

目标：落地 V1 自然语言搜索的完整闭环。

前置依赖：Task 6（账号体系和搜索权限拦截）

范围：

- 新建搜索结果页 `/search`
- 接 `POST /api/v1/search` 接口
- 后端实现问题类型识别与结果路由
- 支持 4 种结果类型：
  - `redirect_compare`：直接跳转对比页
  - `search_result`：搜索结果表格
  - `mixed_result`：结构化结果 + 搜索结果表格
  - `unsupported`：提示"暂不支持该问题类型"
- V1 优先支持的问题类型（PRD 9.4）：
  - 两名球员最近 3 年交手记录
  - 两名球员分别与第三名球员的交手次数、胜率对比
  - 两名球员最近 3 年和某国家对手的交手记录对比
  - 两名球员外战胜率对比
- 搜索接口限流
- 搜索无结果显示"没有找到结果，点击反馈"

附带：

- `POST /api/v1/feedback` 反馈日志接口（PRD 12.3）
