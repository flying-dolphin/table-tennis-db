# ITTF 数据查询产品 UI/UX 设计说明书

版本：v1.0  
日期：2026-04-17

## 1. 设计总纲

### 1.1 设计目标

本产品的 UI/UX 目标不是做资讯站，也不是做炫技型数据大屏，而是成为球迷日常使用的 ITTF/WTT 中文数据查询入口。

设计需要同时满足三件事：

- 查询效率高
- 数据表达清晰
- 移动端浏览舒适

相较官方站点，核心体验优势应体现在：

- 更清晰的信息层级
- 更顺手的跳转路径
- 更友好的中文语境
- 更适合对比和比赛记录浏览的界面结构

### 1.2 设计原则

- 移动端优先：核心页面先按手机尺寸定义，再扩展到桌面端
- 主链路优先：优先服务“排名 -> 选择两名球员 -> 对比”的核心闭环
- 数据即内容：视觉吸引力主要来自排名、统计、交手记录、对战图等数据模块，而不是依赖大图或装饰性内容
- 轻量应用感：整体体验更接近高质量数据工具 App，而非传统门户网站
- 一致口径：同名指标在排名页、球员页、对比页、搜索结构化结果中必须视觉一致、语义一致
- 状态诚实：缺失数据明确显示“暂无数据”，不伪装为 0，不隐藏异常状态

### 1.3 产品气质定位

产品气质定义为：

- 专业，但不冷硬
- 高效，但不后台化
- 数据清晰，但不枯燥
- 有运动感和呼吸感，但不花哨
- 适合长期浏览，而不是只适合快速看数字

界面需要避免以下常见误区：

- 做成传统 BI dashboard，导致界面过于工具后台化
- 只靠表格堆数据，导致浏览体验单调乏味
- 过度装饰和情绪化表达，削弱产品可信度

界面应通过以下方式提升观感与体验：

- 节奏明确的信息层级
- 控制得当的留白
- 局部强调色
- 比赛路径与球员信息卡的视觉重点

### 1.4 视觉方向

视觉母题定义为：`Soft Court Data`

整体方向为：

`移动端优先的数据查询产品 + 运动 editorial 感 + 克制但有活力的界面语言`

视觉基调建议：

- 以浅色背景为主
- 使用深色信息块作为强调层
- 保留圆角卡片和柔和色面
- 通过数据卡片、榜单、对战图和统计模块建立页面吸引力

页面应更像“高质量运动数据卡片流”，而不是“首页 banner + 功能入口拼盘”。

### 1.5 适配当前前端 demo 的保留项与调整项

建议保留：

- 浅底色基调
- 深色头图区
- 圆角卡片
- 悬浮底部导航
- 移动端优先布局

建议调整：

- 首页中偏泛化的英文文案
- 过于空泛的 hero 表达
- 首页信息权重分配
- 底部导航栏目定义
- 列表卡片的数据密度与状态表达

## 2. 信息架构与导航规范

### 2.1 信息架构目标

信息架构设计的目标包括：

- 让用户在最短路径内完成核心查询任务
- 让球员、赛事、比赛、搜索之间形成自然跳转闭环
- 让首页承担入口职责，但不承载过多内容
- 让搜索成为高级能力入口，而不是唯一入口

### 2.2 全局导航结构

产品采用固定底部导航，设置 4 个一级入口：

- 首页
- 排名
- 赛事
- 我的

说明：

- 不单独设置“球员”入口，球员详情页主要从排名页、赛事详情页、比赛详情页、搜索结果页进入
- “我的”在未登录状态下显示登录入口，在已登录状态下显示用户信息入口
- 搜索不进入底部导航，而作为首页主入口和全局高级能力入口存在

### 2.3 首页定位

首页是导航型首页，不是内容门户首页，只保留三个核心模块：

- 搜索框
- 赛事日程
- 世界排名 Top 20

首页职责包括：

- 承接首次访问
- 作为回访用户默认入口
- 为“排名浏览”“赛事浏览”“搜索”提供快速分流

### 2.4 核心页面地图

核心页面包括：

- `/`
- `/rankings`
- `/players/[slug]`
- `/compare`
- `/events`
- `/events/[eventId]`
- `/matches/[matchId]`
- `/search`
- `/auth`

### 2.5 核心主链路

V1 最重要的主链路为：

`首页 -> 排名页 -> 选择两名球员 -> 对比页`

这是设计优先级最高的链路，页面布局、按钮权重、状态反馈都要优先服务这条路径。

### 2.6 次级浏览链路

- 首页 -> 排名页 -> 球员详情页
- 首页 -> 赛事列表页 -> 赛事详情页 -> 比赛详情页
- 首页 -> 搜索结果页 -> 球员页 / 对比页 / 赛事页 / 比赛页
- 球员详情页 -> 比赛详情页
- 赛事详情页 -> 比赛详情页

### 2.7 搜索在架构中的位置

- 搜索是高级能力入口，不是产品主导航入口
- 已登录用户可执行搜索，未登录用户只能看到入口
- 搜索结果优先路由到结构化结果页；当问题无法完全结构化时，再进入搜索结果页承接混合结果

### 2.8 登录在架构中的位置

- 内容浏览链路默认开放
- 搜索链路登录后开放
- 未登录用户点击搜索框时，直接进入登录页
- 登录成功后默认返回首页

### 2.9 架构层面的设计要求

- 当前页面必须有明确 active state
- 页面状态需要支持深链接，尤其是排序、对比对象、赛事年份、`sub event` 切换
- 底部导航、浮层日历、筛选条、对比操作条之间要有清晰层级，不依赖 DOM 顺序碰运气

## 3. 全局布局与页面框架规范

### 3.1 页面框架目标

全局布局的目标不是制造复杂的视觉变化，而是让不同页面在统一骨架下保持：

- 导航清晰
- 信息层级稳定
- 数据模块可复用
- 浏览节奏舒适

所有页面都应让用户快速判断三件事：

- 当前所在位置
- 当前页面的核心任务
- 下一步可执行动作

### 3.2 全局页面骨架

V1 采用移动端优先的应用式页面骨架，基础结构为：

- 顶部页面头部区域
- 中间主内容区域
- 底部固定导航栏

其中：

- 顶部区域负责页面标题、上下文说明、关键操作或状态承接
- 主内容区域负责数据模块、列表、详情内容展示
- 底部导航栏负责全局一级跳转

页面内容必须为底部固定导航预留安全空间，避免主内容被遮挡。

### 3.3 页面容器规范

#### 移动端

- 以单列布局为主
- 使用稳定的左右留白
- 内容宽度不追求铺满屏幕，而应保留呼吸感

#### 桌面端

- 保持移动端信息结构不变
- 在更大屏幕下增加容器宽度
- 避免演变为传统后台式多栏控制台

桌面端的优化目标是“更从容”，而不是“更复杂”。

### 3.4 顶部区域规范

顶部区域分为三类：

#### A. 首页头部

首页头部用于建立产品气质与入口聚焦，应承担：

- 产品名称或当前产品定位说明
- 搜索入口的视觉承接
- 与下方模块形成自然过渡

首页头部不应承担过重的营销文案功能，也不应依赖装饰性大图作为主要信息来源。

#### B. 列表页头部

适用于排名页、赛事列表页、搜索结果页等。

应包含：

- 页面标题
- 当前页面的简短说明
- 关键控制项入口，如排序、年份切换、结果说明

列表页头部要强调“我现在可以浏览什么、切换什么、执行什么”。

#### C. 详情页头部

适用于球员详情页、赛事详情页、比赛详情页、对比页。

应包含：

- 返回路径
- 当前对象名称
- 核心摘要信息
- 当前页的关键操作入口

详情页头部的任务是尽快建立上下文，减少用户向下滚动后才理解页面内容的情况。

### 3.5 内容区模块化规则

主内容区域由模块组成，而不是由零散区块堆叠组成。每个模块应具备：

- 明确标题
- 简短功能说明或辅助信息
- 模块主体内容
- 状态反馈区域

模块之间需要形成稳定节奏：

- 重要模块优先靠上
- 同类模块采用一致结构
- 避免每个模块都使用完全不同的视觉语言

### 3.6 卡片体系分级

V1 建议建立三级卡片体系：

#### 一级卡片：Hero Card

用于页面顶部关键摘要区，例如：

- 首页头部承接区
- 球员详情页顶部信息卡
- 对比页顶部双人信息卡

特点：

- 视觉权重最高
- 可包含深色背景或更强对比
- 用于快速建立页面情绪与核心信息

#### 二级卡片：Data Card

用于统计指标、摘要数字、冠军数、胜率、排名变化等结构化信息。

特点：

- 强调数字与标签的对应关系
- 适合并列展示
- 适合复用于球员页、对比页、搜索结构化结果

#### 三级卡片：List Card

用于列表型信息，如：

- 排名列表
- 最近比赛
- 赛事记录
- 赛事列表
- 搜索结果项

特点：

- 信息密度较高
- 支持点击跳转
- 强调行信息的层级和交互反馈

### 3.7 列表与表格的通用骨架

本产品不应照搬传统后台表格，而应采用“移动端列表优先，桌面端增强信息对齐”的策略。

规则如下：

- 移动端以卡片列表或压缩行列表为主
- 桌面端可增强列对齐，但不强制转为重表头表格
- 交互操作应尽量靠近数据主体，而不是分散在远离内容的位置

适用于排名页、搜索结果页、赛事列表页。

### 3.8 详情页的通用骨架

详情页统一采用以下阅读顺序：

1. 顶部摘要信息
2. 核心统计或关键结果
3. 关键关联内容
4. 扩展浏览内容

例如：

- 球员页：顶部信息 -> 核心统计 -> 最近比赛 -> 赛事记录 -> 常见对手
- 赛事详情页：赛事摘要 -> `sub event` 切换 -> 冠军信息 -> 对战图
- 比赛详情页：比赛摘要 -> 双方信息 -> 比分与局分 -> 返回赛事入口

这样可以保证用户先获得核心事实，再决定是否继续深入浏览。

### 3.9 固定元素与浮层规范

V1 中常见的固定元素和浮层包括：

- 底部导航栏
- 对比操作条
- 月历放大浮层
- 可能的局部筛选条

这些元素必须遵循以下原则：

- 层级清晰
- 关闭路径明确
- 不遮挡核心内容
- 不与页面主操作冲突

在移动端中，浮层不应过多叠加，避免让页面体验变得琐碎和拥挤。

### 3.10 页面节奏与留白原则

为了避免数据产品显得呆板和枯燥，页面节奏必须被明确设计，而不是只把数据模块排满页面。

具体原则：

- 高密度信息区域之后，需要有适当留白或视觉缓冲
- 深色块与浅色块之间要形成节奏变化
- 核心模块要有足够的首屏识别度
- 次要模块应降低噪音，不与主任务争夺注意力

页面的观感应体现“有条理的紧凑”，而不是“压迫式堆叠”。

### 3.11 响应式原则

- 以移动端为默认设计基准
- 平板和桌面端主要优化间距、宽度、模块并列方式
- 不因桌面空间变大而随意增加信息层级
- 核心交互路径在不同尺寸下保持一致

响应式设计的目标是保持体验连续，而不是让不同端像两个产品。

## 4. 视觉系统、主题变量与设计语言规范

### 4.1 视觉系统目标

视觉系统需要服务以下目标：

- 建立清晰可信的产品识别
- 让数据查询过程显得轻盈、耐看、不压迫
- 避免传统数据工具常见的呆板、冰冷和后台感
- 在移动端形成干净、舒适、可长期浏览的阅读体验

视觉层面需要同时实现三件事：

- 用统一的品牌色建立辨识度
- 用稳定的浅色系统保证长期阅读舒适度
- 用局部悬浮感和轻微磨玻璃质感增强页面空气感

### 4.2 整体视觉基调

整体视觉基调定义为：`Sky Editorial Data`

该方向参考品牌 Logo 与浅蓝调视觉样本，形成以下特征：

- 以高明度浅蓝和灰白为主
- 整体氛围轻、净、柔和
- 保持专业秩序感，但避免后台式冷硬感
- 适度引入编辑感和运动感，而不是做纯技术界面

页面气质应接近“浅蓝空气感的数据应用”，而不是：

- 高对比科技风
- 深色数据大屏
- 过度拟物的玻璃风
- 情绪化的体育资讯站

### 4.3 品牌配色方向

#### 主品牌色

品牌主色以浅蓝色为核心：

- `Brand Primary`: `#7FA9D9`

该颜色承担：

- 品牌识别
- 导航和交互激活态
- 关键操作按钮
- 局部视觉强调

#### 扩展品牌色建议

围绕主品牌色建立可扩展色阶：

- `Brand Strong`: `#4F79B3`
- `Brand Deep`: `#6B97CB`
- `Brand Primary`: `#7FA9D9`
- `Brand Soft`: `#DCEAF8`
- `Brand Mist`: `#EAF3FF`
- `Brand Fog`: `#F5F9FE`

这些色阶应覆盖：

- hover / active / selected
- 浅色背景块
- 输入框聚焦态
- 标签底色
- 模块弱强调背景

#### 品牌色分工规则

为了避免品牌浅蓝在高明度页面中“化掉”，品牌色必须分层使用：

- `Brand Primary (#7FA9D9)`：用于品牌识别、氛围色、大面积浅蓝模块、弱强调背景
- `Brand Deep (#6B97CB)`：用于交互激活态、主按钮、active tab、选中边框、聚焦态
- `Brand Strong (#4F79B3)`：用于小面积高重要性交互、文字链接、需要更强对比的小控件状态

规则：

- 小面积交互元素优先使用 `Brand Deep` 或 `Brand Strong`
- `Brand Primary` 不作为唯一的小面积功能色使用
- 交互层的重点判断必须优先依赖对比度稳定的品牌深色阶

### 4.4 页面基础色系统

为了保证高明度浅蓝风格下的数据可读性，页面基础色建议如下：

- `Page Background`: `#F7F9FC`
- `Surface Primary`: `#FFFFFF`
- `Surface Secondary`: `#F8FBFF`
- `Surface Tinted`: `#EEF5FC`
- `Border Subtle`: `#D9E4F2`
- `Border Strong`: `#C7D7EA`
- `Text Primary`: `#1E2A3D`
- `Text Secondary`: `#5F6F86`
- `Text Tertiary`: `#8A9AB0`

使用原则：

- 页面大面积使用浅底和白卡片
- 重要信息通过层级、字重、布局突出，而不是依赖深色块铺满
- 深色仅作为小范围视觉锚点，不作为全站主背景

### 4.5 状态色与功能色

状态色应低噪音、易判断，不破坏浅蓝主基调。

建议如下：

- `Success / Win`: `#4E9B7A`
- `Danger / Loss`: `#C56B72`
- `Warning`: `#C89A4B`
- `Info`: `#6E86A6`

规则：

- 状态色只用于状态判断，不参与品牌主视觉竞争
- 胜负、涨跌、禁用、警示不能只靠颜色表达，应搭配文字、图标或符号
- 所有状态色需优先保证在 `#F7F9FC` 和白色卡片上的可读性
- `Info` 色必须与品牌蓝区分，不可与主品牌色承担相同语义

### 4.5A 颜色对比度校验与修正

本节基于 WCAG 2.1 AA 标准，对全站定义的颜色进行对比度审计，标注不达标项并给出修正方案。

**WCAG AA 标准：**
- 普通文字（< 18px regular / < 14px bold）：最低 **4.5:1**
- 大号文字（≥ 18px regular 或 ≥ 14px bold）：最低 **3:1**
- 非文字交互元素（图标、边框状态指示）：最低 **3:1**

#### 文字色对比度

| 颜色 Token | 色值 | on white | on page-bg | 结论 |
|-----------|------|----------|------------|------|
| `text-primary` | `#1E2A3D` | 14.45:1 | 13.70:1 | 通过 |
| `text-secondary` | `#5F6F86` | 5.12:1 | 4.85:1 | 通过 |
| `text-tertiary` | `#8A9AB0` | 2.87:1 | 2.72:1 | **不达标** |

**修正：`text-tertiary` 从 `#8A9AB0` 改为 `#617283`**

| 颜色 Token | 色值（修正后） | on white | on page-bg | 结论 |
|-----------|-------------|----------|------------|------|
| `text-tertiary` | `#617283` | 4.95:1 | 4.69:1 | 通过 |

修正后的 `#617283` 与原色保持相同色相（冷蓝灰），仅亮度降低，视觉感知差异很小，不影响设计气质。

#### 品牌色作为文字使用时的限制

品牌色作为背景和装饰元素没有约束，但**作为文字颜色**时受对比度限制：

| 颜色 Token | 色值 | on white | 结论 |
|-----------|------|----------|------|
| `brand-primary` | `#7FA9D9` | 2.45:1 | **禁止作为文字色** |
| `brand-deep` | `#6B97CB` | 3.04:1 | 仅限大号粗体文字（≥ 14px bold） |
| `brand-strong` | `#4F79B3` | 4.45:1 | 仅限大号文字（≥ 18px 或 ≥ 14px bold） |

**文字颜色使用规则：**

- `text-brand-primary`：禁止使用，品牌主色亮度过高
- `text-brand-deep`：仅允许用于 `body-lg`（15px）以上的粗体文字，禁止用于 `caption`（12px）和 `micro`（10px）
- `text-brand-strong`：可用于 `body-lg`（15px）bold 及以上；如在 `caption` 大小使用，需加下划线等辅助交互指示
- 小尺寸的链接文字（12px 以下）统一使用 `text-text-primary` 或 `text-text-secondary`，通过下划线区分交互性

**实际影响的修正点：**

| 问题位置 | 当前写法 | 应改为 |
|---------|---------|--------|
| 排名列表"更多"（12px） | `text-text-tertiary hover:text-brand-deep` | `text-text-secondary hover:text-brand-strong` |
| SearchBox "搜索"按钮（12px） | `text-brand-deep` | `text-brand-strong` |
| 对比页"更换"（14px） | `text-brand-deep` | `text-brand-strong` |

#### 状态色作为文字使用时的限制

状态色主要用于背景填充和视觉指示，作为文字时对比度不足：

| 颜色 Token | 色值 | on white | 可用文字尺寸 |
|-----------|------|----------|------------|
| `state-success` | `#4E9B7A` | 3.34:1 | 仅大号文字（≥ 18px 或 ≥ 14px bold）|
| `state-danger` | `#C56B72` | 3.67:1 | 仅大号文字（≥ 18px 或 ≥ 14px bold）|
| `state-warning` | `#C89A4B` | 2.57:1 | **所有尺寸均不达标** |
| `state-info` | `#6E86A6` | 3.73:1 | 仅大号文字（≥ 18px 或 ≥ 14px bold）|

**修正规则：**

- 状态色原色（`#4E9B7A`、`#C56B72`）可用于：背景填充、进度条、图标、大号粗体标签（如胜/负徽标 14px bold）
- **不得**用于 `caption`（12px）和 `micro`（10px）的文字，如排名变化箭头（10px）
- 对于小尺寸胜负/涨跌指示，改用 `text-text-primary` 加符号（↑↓ 或 胜/负），或使用以下定义的深色文字变体：

| 用途 | 文字变体色值 | on white |
|------|------------|----------|
| 胜 / 上升（小文字） | `#3A7A5E` | 5.09:1 |
| 负 / 下降（小文字） | `#9E3E44` | 6.51:1 |
| 警告（小文字） | `#8C5F0A` | 5.59:1 |

这三个深色变体作为 token 扩展在 tailwind.config.ts 中定义：
`state.success-text`、`state.danger-text`、`state.warning-text`

#### 深色背景上的白色文字

| 背景 | 色值 | white on bg | 结论 |
|------|------|-------------|------|
| `hero-anchor` | `#1A232C` | 15.90:1 | 通过，所有尺寸 |
| `brand-strong` | `#4F79B3` | 4.45:1 | 仅大号文字 |
| `brand-deep` | `#6B97CB` | 3.04:1 | 仅大号粗体 |

规则：品牌蓝按钮（`bg-brand-deep`）上的白色文字，字号不得低于 `body-lg`（15px bold）。

### 4.5B 交互无障碍规范

本节定义触控尺寸、焦点状态、动效偏好、颜色语义的最低可访问性要求。

#### 触控目标尺寸

所有可点击元素的可触控区域（包括 padding 扩展后的实际热区）不得小于 **44×44px**（WCAG 2.5.5 建议）。

| 元素 | 要求 | 实现方式 |
|------|------|---------|
| 底部导航每项 | min 44×44px | `min-w-[64px] py-2`（当前已满足） |
| 排名列表行 | min 44px 高度 | `py-2` + 头像高度合计应 ≥ 44px |
| 卡片内小按钮 | min 44×44px 热区 | 必要时用 `p-2` 或 `-m-2 p-2` 扩展热区 |
| 返回按钮 | min 44×44px | `p-2`（24px 图标 + 8px padding × 2 = 40px，需改为 `p-3`） |
| 搜索框 | min 44px 高度 | 当前 `h-14`（56px）已满足 |

不满足时的修复方式：不放大图标，而是增加 padding 扩展热区，必要时用 `min-h-[44px] min-w-[44px]` 配合 `flex items-center justify-center`。

#### 焦点状态

所有可通过键盘 Tab 键聚焦的交互元素，必须有**可见的焦点环**，不得使用 `outline-none` 或 `focus:outline-none` 消除默认焦点环，除非同时定义了替代的焦点样式。

全局焦点样式规范：

```css
/* globals.css 中统一定义 */
:focus-visible {
  outline: 2px solid #6B97CB; /* brand-deep */
  outline-offset: 2px;
  border-radius: 4px;
}
```

- 仅对 `:focus-visible` 生效（键盘触发），不影响鼠标点击后的外观
- 焦点环颜色使用 `brand-deep`，与浅色背景对比度 3.04:1（满足非文字元素 3:1 要求）
- 圆角元素（rounded-full、rounded-lg）的焦点环 `border-radius` 应与元素圆角一致

#### 动效与偏好-减少动效

所有过渡动画和关键帧动画必须遵守 `prefers-reduced-motion` 媒体查询。

```css
/* globals.css 中统一定义 */
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
  }
}
```

具体规则：
- `transition-*` 类：Tailwind 默认不处理此媒体查询，需全局 CSS 覆盖
- `animate-spin`、`animate-pulse`（loading 用）：豁免，因为这些是功能性动画，不是装饰性的
- 大幅位移动画（页面入场 `slide-in`）：受限
- 缩放和淡入动画（`animate-in`）：受限

#### 颜色不是唯一信息载体

所有依赖颜色传达含义的元素，**必须同时提供文字、图标或符号**作为辅助信息。

| 场景 | 禁止 | 要求 |
|------|------|------|
| 排名变化 | 只用红/绿色 | 颜色 + ↑↓ 符号 + 数字 |
| 胜/负结果 | 只用红/绿色 | 颜色 + "胜"/"负"文字 |
| 比赛胜者 | 只用颜色高亮 | 颜色 + 粗体或加亮 + 可选图标 |
| 禁用状态 | 只用灰色 | 灰色 + `disabled` 属性 + opacity |
| 未选/已选 | 只用颜色 | 颜色 + 背景变化 + 文字变化 |

### 4.6 色彩氛围控制原则

为了接近参考图中的轻空气感，同时不牺牲数据产品的清晰度，色彩使用遵循：

- 背景轻
- 内容实
- 强调准
- 层级稳

同一屏建议控制为：

- 1 个基础背景色
- 1 个主品牌色系统
- 1 套中性色文字体系
- 少量状态功能色

不建议引入多个高存在感辅助主色，避免页面显得杂乱。

### 4.7 轻微磨玻璃的使用边界

可以使用轻微磨玻璃和微透明质感，但不能作为全站基础卡片风格。

正确定位为：

- 磨玻璃是“悬浮层语言”
- 实体卡片是“内容层语言”

适合使用轻微磨玻璃的区域：

- 首页搜索框
- 底部悬浮导航
- 对比操作条
- 月历放大浮层
- 少量顶部摘要浮层

不适合使用磨玻璃的区域：

- 排名列表主体(首页排名卡片除外)
- 球员数据主体区
- 对比页核心指标区
- 赛事记录列表
- 搜索结果主体

使用规则：

- 透明度应偏克制，接近高透明白底
- 必须配合清晰边框
- 模糊半径不宜过强
- 不得影响文本对比度和数据稳定性

### 4.8 字体系统

#### 4.8.1 字体选型

中文主字体：`Noto Sans SC / 思源黑体`

选择理由：

- 授权清晰（OFL 开源协议）
- 字形中性、现代、稳定，适合数据产品高频阅读
- 支持 300/400/500/600/700/900 六档字重，满足数据层级需求
- 中英文混排效果好，数字字形清晰

字体加载方式：Google Fonts，使用 `display=swap` 避免 FOIT。

备选方案：`阿里巴巴普惠体`（更具互联网产品气质，正式采用前需核对授权条款）。

不建议在正文区域使用风格强烈的衬线或装饰性中文字体。

#### 4.8.2 字重分工原则

本产品是数据密集型产品，字重是建立信息层级的核心手段。在同一字号下，通过字重变化区分主次，减少对字号跳跃的依赖。

字重语义分配：

| 字重 | 语义 | 典型用途 |
|------|------|---------|
| 900 (Black) | 最高强调 | 产品名、焦点大数字（排名、积分） |
| 700 (Bold) | 强调 | 页面标题、球员姓名、模块标题 |
| 600 (SemiBold) | 中等强调 | 列表中的数据值、卡片标题、次级标题 |
| 500 (Medium) | 弱强调 | 标签文字、辅助链接、排名数字 |
| 400 (Regular) | 默认 | 正文描述、说明文字、长文本 |

规则：

- 密集数据区（排名列表、对比指标）主要靠字重分层，不靠字号分层
- 焦点展示区（Hero、数据卡片）可同时使用大字号 + 高字重建立视觉重心
- 同一个组件内字重跨度不超过 3 档（如 400/600/700），避免对比失控

### 4.9 字号系统

#### 4.9.1 字号标尺

全站使用 8 级字号标尺，消除随意的中间值（如 11px、13px、19px）。

| Token | 移动端 | 桌面端 | 行高 | 字间距 | 用途 |
|-------|--------|--------|------|--------|------|
| `display` | 28px | 32px | 1.1 | -0.01em | 产品名（首页 Hero） |
| `data-hero` | 24px | 28px | 1.15 | -0.01em | 卡片焦点数字（排名 #3、积分 8200、胜率 78.3%） |
| `heading-1` | 20px | 22px | 1.25 | -0.01em | 页面标题、球员姓名（Hero 区） |
| `heading-2` | 17px | 18px | 1.3 | 0 | 模块标题（核心统计、最近比赛、交手记录） |
| `body-lg` | 15px | 16px | 1.4 | 0 | 列表中的球员名、关键数据值、卡片标题 |
| `body` | 14px | 14px | 1.5 | 0 | 默认正文、描述、赛事名称 |
| `caption` | 12px | 12px | 1.4 | 0.01em | 日期、标签、辅助链接、国家名称 |
| `micro` | 10px | 10px | 1.3 | 0.03em | 单位后缀（分）、导航标签、徽标、Tags |

#### 4.9.2 设计决策说明

**为什么新增 `data-hero` 层级：**

本产品核心场景是数据卡片 — 排名、积分、胜率、冠军数等数字需要一个专属的"大数字"层级。`data-hero`（24px / 700-900）在卡片内形成焦点，同时不会让卡片头重脚轻。不再使用 28-32px 的过大数字。

**为什么 `body` 是 14px 而不是 16px：**

通用 Web 规范建议 16px 最小正文，但该结论基于拉丁字母。中文字符视觉面积比同字号拉丁字母大约 20-30%，14px 中文的阅读体验约等于 16px 英文。数据列表场景需要信息密度，14px 是密度和可读性的最佳平衡点。

**为什么消灭 11px 和 13px：**

11px 夹在 micro(10px) 和 caption(12px) 之间，13px 夹在 caption(12px) 和 body(14px) 之间，它们不贡献任何层级区分，只增加视觉噪音。11px 内容归入 caption 或 micro；13px 内容统一提到 body。

#### 4.9.3 各层级的典型使用场景

**`display`（28px）：**

- 首页 Hero 区产品名"豆包球谱"
- 全站仅此一处使用

**`data-hero`（24px）：**

- 球员详情页：当前排名、当前积分
- 对比页：H2H 胜场数
- 球员详情页 Bento 卡片：生涯总胜率、赛事总数、冠军数
- 字重范围 700-900，视卡片重要性选择

**`heading-1`（20px）：**

- 页面标题：世界排名、球员对比
- 球员 Hero 区姓名
- 排名列表第 1 名的排名数字
- 字重固定 700

**`heading-2`（17px）：**

- 模块标题：职业看板、最近比赛、比赛记录、Top 3 对手
- 对比页 sticky header 标题
- 字重 600 或 700

**`body-lg`（15px）：**

- 排名列表中的球员中文名
- 对比页核心指标数值
- 卡片标题（赛事名、对手名）
- 交手记录中的比分
- 字重 500-700，视内容角色选择

**`body`（14px）：**

- 默认正文和说明文字
- 赛事名称、轮次信息
- 搜索框 placeholder
- 空状态提示文字
- 字重 400 默认，600 用于强调

**`caption`（12px）：**

- 日期和时间信息
- 国家名称（如"中国""日本"）
- 模块 hint（如"按赛事倒序，不逐场展开"）
- 辅助链接（"更多""更换"）
- 字重 400-600

**`micro`（10px）：**

- 数字单位后缀："分""场""次"
- 底部导航标签
- Bento 卡片内的指标名称标签
- 球员 Hero 区的信息 Tags（性别、年龄、打法）
- 排名变化箭头
- 字重 500-600，小字必须有足够字重保证可读
- 字间距 0.03em，加宽补偿小字号的紧凑感

### 4.10 数字排版专项规范

数据产品中，数字是用户扫读的核心内容。数字排版需要独立规范，不能套用文字排版的默认规则。

#### 4.10.1 等宽数字

所有数据数字（积分、胜率、比分、排名、冠军数、交手次数）必须使用等宽数字特性：

```css
font-variant-numeric: tabular-nums;
```

等宽数字保证列表中的数字列严格对齐，不因字形宽度差异（如 1 比 8 窄）导致视觉抖动。

在 Tailwind 中使用 `tabular-nums` 工具类。

#### 4.10.2 数字字重规则

| 场景 | 字重 | 示例 |
|------|------|------|
| 卡片焦点数字 | 700-900 | 排名 #3、积分 8,200、胜率 78.3% |
| 列表内数据 | 600 | 排名列表中的积分、对比页指标值 |
| 辅助数字 | 500 | 排名变化（↑2）、年份 |

#### 4.10.3 数字附属元素

- 百分比符号 `%` 与数字同级同重，不单独缩小
- 单位后缀（"分""场""次"）使用 `micro` 层级（10px），与数字间距 2px
- 排名前缀 `#` 与排名数字同级同重
- 比分分隔符 `:` 与比分数字同级同重

#### 4.10.4 涨跌与胜负标记

- 排名上升：`state-success` 色 + "↑" + 数字
- 排名下降：`state-danger` 色 + "↓" + 数字
- 比赛胜：`state-success` 色
- 比赛负：`state-danger` 色
- 不依赖纯颜色传达含义，必须搭配文字或箭头符号

### 4.11 间距、圆角、阴影与玻璃效果

#### 4.11.1 间距系统

本产品采用 4px 基准栅格。所有间距值必须是 4px 的整数倍，不使用半步值（如 2.5、3.5）。

**允许的间距值：**

| Tailwind 类 | 实际值 | 语义角色 |
|-------------|--------|---------|
| `1` | 4px | 最小间距：图标与文字的行内间隙 |
| `2` | 8px | 紧凑间距：组件内部元素间隙、标签组 |
| `3` | 12px | 默认间距：卡片内部分区、列表项内边距 |
| `4` | 16px | 标准间距：卡片内边距、模块标题到内容 |
| `5` | 20px | 页面间距：所有页面统一水平内边距 |
| `6` | 24px | 模块间距：页面内相邻模块之间的垂直距离 |
| `8` | 32px | 大间距：页面重要分区之间 |
| `12` | 48px | 特大间距：仅用于页面顶部/底部安全区 |

**不允许使用的值：**

- `0.5`（2px）、`1.5`（6px）、`2.5`（10px）、`3.5`（14px）：半步值破坏栅格节奏
- `7`、`9`、`11` 等非常规值：无语义，增加维护成本

**语义化间距规则：**

| 场景 | 间距 | 说明 |
|------|------|------|
| 页面水平内边距 | `px-5`（20px） | 全站统一，不再混用 px-4/px-5/px-6 |
| 模块之间垂直间距 | `pt-6`（24px） | 统一模块节奏感 |
| 卡片内边距 | `p-4`（16px） | 标准内容卡 |
| 紧凑卡片内边距 | `p-3`（12px） | 列表项、密集数据卡 |
| 卡片之间间距 | `gap-3`（12px） | 堆叠卡片的统一间隙 |
| 模块标题到内容 | `mb-3`（12px） | Section Header 下方 |
| 底部导航安全区 | `pb-20`（80px） | 避免内容被悬浮导航遮挡 |
| Bento 卡片间距 | `gap-3`（12px） | Bento Grid 内部 |

**间距与信息密度的关系：**

数据产品中间距不是"越大越好"。不同区域的间距策略不同：

- **焦点区（Hero、搜索框）**：间距宽松，建立呼吸感，使用 `space-6` 到 `space-8`
- **浏览区（排名列表、赛事列表）**：间距紧凑但不拥挤，使用 `space-2` 到 `space-3`
- **数据卡片内部（Bento、对比指标）**：间距克制，标签与数字之间 `space-1`，分区之间 `space-3`

#### 4.11.2 圆角系统

全站使用 4 级圆角标尺 + `full` 特殊值，消除 22px、26px、32px、34px、40px 等随意中间值。

| Token | 值 | Tailwind | 适用对象 |
|-------|-----|----------|---------|
| `radius-sm` | 8px | `rounded-lg` | 按钮、输入框、Tags、徽标、小控件 |
| `radius-md` | 16px | `rounded-2xl` | 数据卡片、Bento 卡片、列表项、搜索框 |
| `radius-lg` | 24px | `rounded-3xl`（需配置） | 模块容器、弹窗、赛事卡、最近比赛卡 |
| `radius-full` | 9999px | `rounded-full` | 头像、胶囊按钮、底部导航、圆形指示器 |

**迁移映射：**

| 当前值 | 归属 | 说明 |
|--------|------|------|
| 3px、6px | `radius-sm`（8px） | 过小的圆角统一提升 |
| 22px | `radius-md`（16px） | Bento 卡片不需要这么大 |
| 24px | `radius-lg`（24px） | 保持不变 |
| 32px、34px、40px | `radius-lg`（24px） | 过大值收敛到 24px |
| full | `radius-full` | 保持不变 |

**规则：**

- 同一屏内只允许出现 2-3 种圆角等级，不得每个组件一个半径
- 嵌套组件的内层圆角应小于外层（外层 `radius-lg`，内层 `radius-md` 或 `radius-sm`）
- Hero 区底部圆角使用 `radius-lg`（24px），不再使用 34px 等非标准值
- Tailwind 配置中 `borderRadius['3xl']` 应设为 `24px`

#### 4.11.3 阴影系统

全站使用 4 级阴影标尺，统一阴影颜色基底为 `rgba(30, 42, 61, alpha)`（与 `Text Primary` 同色系），保证阴影色调与品牌浅蓝基调协调。

| Token | CSS 值 | 用途 |
|-------|--------|------|
| `shadow-subtle` | `0 1px 3px rgba(30,42,61,0.06)` | 卡片静息态、列表项默认态 |
| `shadow-card` | `0 4px 12px rgba(30,42,61,0.08)` | 交互卡片、hover 提升态 |
| `shadow-float` | `0 12px 32px rgba(30,42,61,0.12)` | 悬浮元素（对比操作条、搜索聚焦态） |
| `shadow-nav` | `0 20px 45px rgba(30,42,61,0.16)` | 底部悬浮导航（全站最强阴影） |

**规则：**

- 不再使用 Tailwind 默认的 `shadow-sm`/`shadow-md`/`shadow-lg`/`shadow-xl`/`shadow-2xl`，统一使用语义化 token
- 不再使用 `shadow-inner`，active 状态通过背景色变化而非内阴影表达
- 卡片 hover 时从 `shadow-subtle` 过渡到 `shadow-card`，配合 `transition-shadow duration-200`
- 模态遮罩层不用阴影，而是用半透明背景 `bg-black/35` + `backdrop-blur-sm`

#### 4.11.4 玻璃效果与透明度系统

当前代码中存在 13 种不同的 `bg-white/XX` 值，必须收敛。

**浅色玻璃卡片透明度（浅色背景上使用）：**

| Token | 值 | 用途 |
|-------|-----|------|
| `glass-solid` | `bg-white/85` | 高可读性内容卡（搜索框、弹窗内容区） |
| `glass-medium` | `bg-white/70` | 默认玻璃卡片（排名容器、赛事卡） |
| `glass-light` | `bg-white/50` | 悬浮层背景、hover 提示区 |

**深色背景上的透明层（Hero 区、深色 Hero 卡片内使用）：**

| Token | 值 | 用途 |
|-------|-----|------|
| `on-dark-card` | `bg-white/14` | 深色区内的数据卡片（排名卡、积分卡） |
| `on-dark-hint` | `bg-white/10` | 深色区内的辅助装饰 |

**不允许使用的值：**

white/12、white/15、white/20、white/30、white/40、white/45、white/55、white/60、white/75、white/80、white/90 — 这些中间值全部归入最近的标准值。

**模糊强度（backdrop-blur）：**

| Token | 值 | 用途 |
|-------|-----|------|
| `blur-card` | `backdrop-blur-md`（12px） | 玻璃卡片 |
| `blur-float` | `backdrop-blur-xl`（24px） | 悬浮导航、模态遮罩 |

仅使用这两档。不使用 `backdrop-blur-sm`（太弱无效果）和 `backdrop-blur-2xl`（过度模糊）。

#### 4.11.5 边框规范

**颜色规则：**

- 浅色卡片边框统一使用 `border-border-subtle`（`#D9E4F2`）
- 需要更强边界时使用 `border-border-strong`（`#C7D7EA`）
- 激活态 / 选中态使用 `border-brand-deep`
- 不使用 `border-white/XX` 的透明白色边框 — 它在浅色背景上几乎不可见
- 深色背景上的卡片边框使用 `border-white/14`

**宽度规则：**

- 默认 `border`（1px）
- 不使用 `border-2` 及以上，避免厚重感

#### 4.11.6 视觉结构 Token 汇总

### 4.12 z-index 层级规范

#### 4.12.1 层级标尺

全站使用 6 级 z-index 标尺，每级含义明确，不使用随意数字（如 z-[70]、z-[100]）。

| Token | 值 | Tailwind | 用途 |
|-------|-----|----------|------|
| `z-above` | 10 | `z-10` | 局部堆叠上下文：相对定位的装饰层、卡片内图标遮盖 |
| `z-sticky` | 30 | `z-30` | 页面 sticky 头部（排名页、对比页顶部导航条） |
| `z-bar` | 40 | `z-40` | 悬浮操作条（对比操作条，位于底部导航上方） |
| `z-nav` | 50 | `z-50` | 底部悬浮导航（全局固定，始终可见） |
| `z-overlay` | 60 | `z-60` | 所有模态遮罩和弹窗（登录提示、日历放大、局部说明层） |

`z-above` 到 `z-overlay` 之间留有足够间隔，确保各层不会意外交叉。

#### 4.12.2 关键决策说明

**为什么消灭 z-[70] 和 z-[100]：**

原代码中登录弹窗用 z-[70]、日历弹窗用 z-[100]，这是"弹窗叠弹窗"的防御性写法。在本产品中，任意时刻只会有一个模态层出现（互斥设计），因此所有模态统一用 `z-overlay`（60）即可，无需相互竞争。

**为什么没有 z-20：**

当前代码中 z-20 只有搜索框容器在用，目的是让搜索框浮于 Hero 图片之上。这是局部上下文问题，应通过正确的 DOM 结构（搜索框渲染在 Hero 之后）解决，不需要独立的 z-index 层级。

#### 4.12.3 使用规则

- 只使用上表中定义的 5 个值，**禁止**使用其他任意数字
- `z-above` 仅用于同一容器内的局部层叠，不用于跨组件竞争
- 需要新增层级时，先评估是否能通过 DOM 顺序或局部 stacking context 解决
- 负值 `-z-10` 仅用于同一容器内的背景装饰层，不得影响页面主内容

#### 4.12.4 各元素层级归属

| 元素 | 层级 | 说明 |
|------|------|------|
| 卡片内装饰背景、图标 | `z-above` | 局部上下文 |
| 排名页 sticky 头部 | `z-sticky` | 随滚动固定 |
| 对比页 sticky 头部 | `z-sticky` | 随滚动固定 |
| 对比操作条 | `z-bar` | 固定于底部导航上方 |
| 底部导航 | `z-nav` | 全局最高固定层 |
| 登录弹窗 | `z-overlay` | 互斥，只同时出现一个 |
| 日历放大浮层 | `z-overlay` | 互斥，只同时出现一个 |
| 局部说明弹层 | `z-overlay` | 互斥，只同时出现一个 |

### 4.13 图标、插图与视觉元素

- 全站统一使用 SVG 图标系统
- 不使用 emoji 作为正式 UI 图标
- 图标风格应简洁、线性、统一
- 装饰图形可少量引入圆弧、轨迹、柔和渐变等与运动相关的抽象元素

视觉辅助元素应服务”轻、净、柔和”的整体氛围，不应做成炫技装饰。

### 4.14 动效语言

动效应强调轻盈和顺滑，避免“科技炫动感”。

重点场景包括：

- 页面进入的轻量渐入
- 排序切换和列表重排
- 浮层开合
- 卡片 hover / press
- 对比条激活反馈

规则：

- 动效时长短而克制
- 不使用夸张位移和大幅缩放
- 不影响数据读取效率
- 必须兼容 `prefers-reduced-motion`

### 4.15 数据表达的视觉原则

数据页面必须有清晰视觉语法，避免信息“铺平”。

建议统一：

- 排名具备高识别样式
- 核心数字拥有固定位置与固定表现
- 涨跌、胜负、状态有统一标记方式
- 列表中最关键字段优先被扫读到

页面应让用户在短时间内判断：

- 核心对象是谁
- 当前最关键的数据是什么
- 下一步可以点击哪里

### 4.15A 视觉细节校准规则

为了让首页和核心页面更接近“轻、净、软、悬浮、年轻”的目标效果，视觉落地时需要额外遵循以下细节规则。

#### 留白比例与首屏密度

- 首屏必须克制信息量，不追求把关键内容一次性塞满
- 页面应优先建立呼吸感，再承接数据内容
- 模块之间要有明确留白，不能只靠细边框分隔
- 首屏建议控制为“一个主焦点 + 一个次焦点 + 一个弱提示”，避免三个以上高权重模块同时争抢注意力

#### 卡片密度与白卡占比

- 首页和轻内容模块应以大面积白卡为主，而不是大量小卡片堆叠
- 卡片内部信息层级要少而准，避免每张卡装入过多字段
- 不同卡片之间应保留足够间距，让页面保持轻盈
- 高密度数据内容应延后出现在首屏以下区域

#### 圆角等级细化

严格遵循 4.11.2 定义的 4 级圆角标尺：

- Hero Card、模块容器、弹窗：`radius-lg`（24px）
- 数据卡片、Bento 卡片、列表项：`radius-md`（16px）
- 按钮、输入框、Tags：`radius-sm`（8px）
- 头像、底部导航、胶囊按钮：`radius-full`
- 同一屏内只允许出现 2-3 种圆角等级

#### 深色锚点比例

- 页面中需要保留少量高对比深色锚点，用于建立视觉重点
- 深色块不应大面积滥用，而应少而准
- 首页首屏建议只保留一个主深色锚点，或一个主深色锚点加一个弱深色辅助块
- 深色锚点可用于按钮、关键摘要数字、底部导航中心按钮等位置

#### 底部导航造型倾向

- 底部导航不应做成传统扁平工具栏
- 更适合采用悬浮式、圆角化、具备轻雕塑感的形态
- 如存在中心主操作按钮，应使其在体量、层级和交互反馈上明显区别于普通导航项
- 整体要像“可触摸的物件”，而不是一条普通功能条

#### 首页排名模块的露出方式

- 首页排名模块不应在首屏过早打断轻盈气质
- 排名列表可以作为首页的主要数据模块，但不宜一开始就出现完整工具榜单感
- 首次露出时更适合采用较克制的卡片列表或摘要列表形式
- 更完整、更强功能性的榜单体验应进入排名页承接

#### 视觉完成度的判断标准

如果页面出现以下情况，说明实现结果偏离目标气质：

- 首屏信息过满，像功能列表页
- 所有卡片大小接近，缺少节奏
- 所有交互都只靠浅蓝，没有深色锚点
- 底部导航过于普通，缺少悬浮感和识别度
- 首页排名模块一出现就让页面回到“数据工具首页”的印象

达到目标效果的判断标准应是：

- 看起来轻，但不空
- 看起来柔和，但重点仍然清楚
- 看起来像消费级移动产品，而不是管理后台
- 即使有数据模块，页面仍然有空气感和年轻感

### 4.16 文案语气与界面语言

界面文案要求：

- 中文优先
- 简洁直接
- 清楚自然
- 有人格感但不油腻
- 不使用公文腔和过度营销化表达

统一用语包括：

- `暂无数据`
- `没有找到结果`
- `暂不支持该问题类型`
- `点击反馈`

### 4.17 主题变量与实现可配置原则

配色、字体和视觉细节必须在实现层面设计为可配置、可替换、可集中管理的主题系统。

实现要求包括：

- 所有核心颜色通过 design tokens 统一定义
- 所有字体通过全局主题变量统一配置
- 圆角、边框、阴影、透明度、模糊强度等应抽象为可复用变量
- 组件中不应直接散落硬编码颜色和字体
- 页面级样式应优先引用语义化 token，而不是直接写十六进制值

建议的 token 类型包括：

- `color.brand.*`
- `color.surface.*`
- `color.text.*`
- `color.state.*`
- `font.family.*`
- `font.size.*`
- `radius.*`
- `shadow.*`
- `blur.*`
- `opacity.*`

这样可以保证：

- 后续更换主色成本低
- 调整中文主字体成本低
- 浅蓝方案可快速演化为其他主题
- 前端实现和设计规范保持一致

## 5. 组件系统与交互组件规范

### 5.1 组件系统目标

组件系统的目标不是单纯提高复用率，而是让产品在不同页面中保持：

- 一致的交互逻辑
- 一致的视觉层级
- 一致的状态表达
- 可扩展、可维护的实现结构

组件系统应优先支撑以下高频场景：

- 导航
- 搜索
- 排名浏览
- 双球员对比
- 数据摘要
- 赛事与比赛浏览
- 空状态与异常状态反馈

### 5.2 组件分层原则

建议将组件分为三层：

#### 基础层

承担通用视觉与交互能力，例如：

- Button
- Input
- Chip
- Tag
- Card
- Modal / Drawer
- Tabs
- Avatar
- Icon Button

#### 业务层

承担与产品语义相关的组合模块，例如：

- Bottom Navigation
- Search Entry
- Ranking Row
- Compare Action Bar
- Data Stat Card
- Event Calendar Card
- Match Row
- Empty State Card

#### 页面层

承担特定页面的编排与组合逻辑，例如：

- Home Ranking Section
- Compare Metrics Section
- Player Recent Matches Section
- Event Bracket Section

规则：

- 页面层优先组合业务层组件
- 业务层优先组合基础层组件
- 避免页面中直接重复拼装基础交互而绕过业务层语义

### 5.2A 组件来源与实现策略

组件体系不应先被第三方 UI 库定义，而应先由产品语义定义，再选择合适的实现方式。

实现策略建议如下：

- 基础原子组件可优先使用成熟开源组件能力
- 业务语义组件应由项目自行组合和定义
- 页面结构与交互规则必须以产品设计说明书为准，而不是以第三方组件命名为准

推荐策略：

- 使用开源组件承接基础能力，例如按钮、输入框、对话框、抽屉、checkbox、tabs、tooltip
- 使用项目自定义组件承接业务语义，例如 Ranking Row、Compare Action Bar、Event Calendar Card、Bracket Section、Match Row
- 对于开源组件库中没有现成定义的模式，应由项目基于基础组件自行组合实现

如果使用 `shadcn/ui`，其定位应是：

- 作为基础层和交互原子层的实现加速工具
- 不直接决定业务组件结构
- 不替代产品中的语义化组件命名与规范

### 5.3 全局导航组件

#### 底部导航栏

底部导航栏是 V1 的一级主导航组件，应满足：

- 固定显示
- 当前页 active state 明确
- 与页面底部留白系统联动
- 在移动端拥有足够点击面积

组件结构包括：

- 图标
- 标签文字
- 当前态反馈

交互规则：

- 当前页必须明确高亮
- 未激活项保持低对比但可识别
- 不通过仅颜色变化表达当前态，应同时配合背景、字重或图标强调
- 未登录状态下，“我的”入口承担登录入口角色

视觉建议：

- 使用轻微磨玻璃或微透明悬浮层
- 使用 `Brand Deep` 作为 active state 核心色
- 保持圆角和空气感，但不能削弱识别度

### 5.4 搜索入口组件

搜索入口是首页最重要的主交互组件，应区别于普通输入框。

组件任务：

- 承接自然语言搜索入口
- 在未登录状态下承接登录引导
- 在视觉上成为首页首个交互焦点

结构建议：

- 搜索图标
- 输入提示文本
- 示例问题或辅助说明
- 可选的功能按钮或状态提示

交互规则：

- 未登录：可见但不可执行，点击后直接进入登录页
- 已登录：可输入并跳转搜索结果页
- 输入区域须保持高可点击性，不能只剩极细文本点击区域

视觉建议：

- 可采用轻微磨玻璃质感
- 与首页头部形成一体化视觉承接
- 占据足够首屏权重，但不盖过页面标题

### 5.5 按钮系统

按钮应至少分为以下层级：

- `Primary Button`
- `Secondary Button`
- `Ghost Button`
- `Danger Button`
- `Text Button`

规则：

- 主按钮使用 `Brand Deep`
- 次按钮优先使用浅底 + 边框方案
- 危险操作使用独立 danger 色
- 文本按钮仅用于低风险、低权重操作
- 不同层级按钮不可只靠尺寸区分，必须有清晰视觉差异

状态要求：

- Default
- Hover
- Pressed
- Disabled
- Loading

### 5.6 输入框与筛选项

输入类组件包括：

- 普通输入框
- 搜索输入框
- 排序切换项
- 年份切换项
- `sub event` 切换项
- 勾选项 / checkbox

规则：

- 默认状态下边界清晰
- 聚焦态使用 `Brand Deep`
- 禁用态明确但仍可辨认
- 选中态与 hover 态需明显区分

对于排序、年份、`sub event` 这类选择组件，建议统一采用 Chip / Segmented Control 语言，而不是每页自定义一种样式。

### 5.7 卡片系统

卡片系统承接第 3 章定义的三级结构，组件层面需明确可复用规范。

#### Hero Card

用于：

- 首页头部摘要
- 球员页顶部信息区
- 对比页顶部摘要区

要求：

- 视觉权重最高
- 支持头像、标题、辅助信息、重点数字组合
- 可接受更强背景层次

#### Data Card

用于：

- 排名
- 积分
- 胜率
- 冠军数
- 七大赛进决赛次数

要求：

- 标签与数值关系稳定
- 数字易扫读
- 支持正负变化、状态说明、趋势标记

#### List Card

用于：

- 排名列表项
- 比赛记录
- 最近比赛
- 赛事列表
- 搜索结果项

要求：

- 高密度但不拥挤
- 点击反馈明确
- 信息主次分明

### 5.8 排名列表项组件

排名列表项是产品中最高频的业务组件之一。

必须支持：

- 排名展示
- 球员头像
- 中文名 / 英文名
- 国家
- 积分
- 排名变化
- 当前排序字段值
- 勾选用于对比

交互规则：

- 点击球员主体区域进入球员详情页
- 勾选区域独立可操作，不应误触详情跳转
- 已勾选状态必须有明确反馈
- 达到两人后激活对比操作条
- 第三人不可继续勾选

视觉重点：

- 排名与球员身份信息优先被看见
- 数值区保持对齐和稳定
- 勾选行为不能破坏列表节奏

### 5.9 对比操作条组件

对比操作条是承接核心主链路的关键组件。

出现时机：

- 排名页勾选 1 人后出现弱状态
- 勾选 2 人后进入强激活状态

组件应显示：

- 已选球员数量
- 已选球员简要信息
- 发起对比按钮
- 更换或取消选择入口

视觉建议：

- 使用悬浮式组件
- 可采用轻微磨玻璃
- 激活状态用 `Brand Deep` 明确强调

交互要求：

- 2 人时按钮明确可点击
- 少于 2 人时提供状态提示
- 不应遮挡底部导航核心区域

### 5.10 数据指标卡组件

数据指标卡用于承接结构化统计结果，应统一于：

- 球员详情页
- 对比页
- 搜索结构化结果

建议字段结构：

- 指标名称
- 数值
- 单位或辅助标签
- 补充说明或状态信息

要求：

- 同名指标样式一致
- 数值的视觉权重大于标签
- 不同卡片之间排布节奏稳定

### 5.11 赛事与比赛列表组件

赛事和比赛相关组件应严格区分 `event`、`sub event`、`match` 三个层级。

#### 赛事列表项

应展示：

- 赛事名称
- 时间
- 年份信息
- 入口状态

#### `sub event` 区块组件

`sub event` 是赛事详情页中的核心层级，应作为独立组件定义，而不能混入 `event` 或 `match` 结构中。

应承担：

- `sub event` 切换入口
- 当前 `sub event` 的冠军摘要
- 对战图主体
- 晋级路径浏览
- 无数据状态反馈

建议拆分为以下子组件：

- `Sub Event Switcher`
- `Champion Summary Card`
- `Bracket Section`
- `Bracket Match Node`

其中：

- `Sub Event Switcher` 用于多个赛事项目之间的互斥切换
- `Champion Summary Card` 用于展示当前项目冠军与摘要信息
- `Bracket Section` 用于承接完整正赛对战图
- `Bracket Match Node` 用于表示对战图中的单场比赛节点

交互规则：

- 普通赛事默认进入女单
- 团体赛默认进入女子团体
- 无数据的 `sub event` 入口需置灰，但保留存在
- 切换 `sub event` 后，页面应更新冠军摘要与对战图主体
- 对战图节点点击后进入比赛详情页

视觉要求：

- `sub event` 入口应明显区别于普通标签
- 冠军摘要区与对战图区需形成清晰层级
- 对战图节点要保证在移动端仍可点击和辨认

#### 比赛列表项

应展示：

- 赛事名
- 轮次 / 阶段
- 对手
- 比分
- 结果

要求：

- 赛事层与比赛层视觉结构不能混淆
- 比赛结果信息需要快速识别
- 跳转入口必须明确

### 5.12 Tabs、Chips 与切换控件

以下组件需统一成同一套交互语言：

- 排名排序切换
- 赛事年份切换
- `sub event` 切换
- 登录 / 注册切换

建议规则：

- 高并列、互斥、同层级视图切换使用 Segmented Tabs
- 轻量筛选、标签式控制、弱操作入口使用 Chips
- 当前选中态必须明显
- 置灰态需要保留可见性

语义区分如下：

#### Segmented Tabs

适用于：

- 排名排序切换
- `sub event` 切换
- 登录 / 注册切换

特征：

- 选项数量通常较少
- 选项彼此互斥
- 所有选项处于同一层级
- 当前选中项直接决定当前视图或当前数据上下文

#### Chips

适用于：

- 年份快速切换
- 辅助筛选
- 轻量标签状态

特征：

- 语义比 Tabs 更轻
- 可以是单选，也可以扩展为多选
- 更偏向筛选或标签控制，而不是页面主视图切换

对于无数据的 `sub event`，入口应置灰但保留存在感，不直接隐藏。

### 5.13 状态标签与结果标记

状态标签应统一用于：

- 胜 / 负
- 排名升 / 降
- 当前选中
- 已登录 / 未登录
- 支持 / 不支持

规则：

- 颜色、边框、文字三者协同表达
- 不依赖单一颜色传达意义
- 标签尺寸与字重保持统一

### 5.14 空状态、缺失状态与异常状态组件

状态反馈组件必须标准化，避免每页独立发挥。

应覆盖：

- `暂无数据`
- `没有找到结果`
- `暂不支持该问题类型`
- `点击反馈`
- `已反馈`
- 简单 loading

规则：

- 空状态文案统一
- 空状态样式统一
- 提供明确下一步动作
- 缺失数据不伪装成 0

### 5.15 浮层、弹窗与放大查看组件

适用场景包括：

- 月历放大态
- 登录承接层
- 局部说明弹层

要求：

- 层级明确
- 开合动线清楚
- 背景遮罩克制
- 关闭方式明确
- 不在移动端堆叠过多层级

可使用轻微磨玻璃，但内容容器仍需保持足够实体感和清晰边界。

### 5.16 组件实现与主题映射要求

组件系统在实现中必须直接映射第 4 章主题变量，而不是各自硬编码。

要求：

- 组件颜色引用语义化 token
- 组件字体引用全局字体 token
- 组件圆角、阴影、边框、透明度、模糊强度引用统一 token
- 业务组件不得绕过基础组件体系直接定义一套新风格

这样可以保证：

- 后续统一换肤成本低
- 设计规范与代码实现一致
- 页面扩展时风格不会逐步失控

## 6. 产品人格与文案语气规范

### 6.1 人格定位

本产品不是冷冰冰的数据工具，而是一个有明确性格的数据型陪看伙伴。

人格方向定义为：

- 活泼
- 开朗
- 大方
- 机灵
- 懂球
- 有点态度
- 不端着

参考气质：

- 更接近孙颖莎式的轻快、松弛、直给
- 有亲和力，但不撒娇
- 有一点“会回嘴”的劲儿，但不过界
- 让用户觉得这是“自己人”，不是官方公告栏

### 6.2 语气总原则

文案整体应满足：

- 先清楚，再有趣
- 先自然，再有梗
- 先像人说话，再像系统提示

目标不是让每一句话都很有风格，而是让关键交互节点更有人味、更有记忆点。

### 6.3 适合活泼化的场景

以下场景可以明显增加人格感：

- 搜索引导文案
- 空状态文案
- 无结果提示
- 反馈成功提示
- 登录引导文案
- 对比入口提示
- 最近比赛缺失提示
- 搜索不支持问题类型提示

这些场景可以让产品更像在和用户对话，而不是机械报错。

### 6.4 不适合过度活泼的场景

以下场景必须保持专业、稳定、准确：

- 排名
- 积分
- 胜率
- 比分
- 对战记录
- 赛事事实信息
- 登录 / 注册错误提示
- 权限限制
- 数据缺失说明中的关键事实

原则：

- 数据内容本身不卖萌
- 系统引导和状态反馈可以更有性格

### 6.5 “略微怼人”的边界

产品可以有一点俏皮、直给、带轻微吐槽感，但不能：

- 阴阳怪气
- 冒犯用户
- 责怪用户
- 用过强的粉圈黑话排斥普通用户
- 把严肃错误写成玩笑

允许的表达方向包括：

- `这题有点难接`
- `先别急`
- `这场我还没翻出来`
- `你这个问法挺会挑重点`

不允许的表达方向包括：

- `这你都不会`
- `你问得也太离谱了`
- `自己去看`
- 任何让用户感到被嘲讽的表达

### 6.6 文案风格规则

建议使用：

- 短句
- 口语化表达
- 有节奏感的中文语序
- 轻微拟人化表达

避免使用：

- 公文腔
- 术语堆叠
- 过长解释
- 强行网络热梗
- 过度可爱化语气词

不建议频繁使用：

- `呀`
- `啦`
- `呢`
- 过度感叹号
- 只有粉丝才懂的高门槛缩写和内部梗

### 6.7 示例文案方向

以下为从标准工具风转向轻人格风的示例：

- `暂无数据` -> `这条还没收进来`
- `没有找到结果` -> `这次没翻到你要的结果`
- `暂不支持该问题类型` -> `这个问题有点刁钻，当前版本还接不住`
- `点击反馈` -> `点我催更`
- `已反馈` -> `收到，记下了`
- `未登录不可搜索` -> `想搜这题，先登录一下`
- `请选择两名球员进行对比` -> `先选两位，我们再开比`
- `最近比赛暂无数据` -> `她最近这几场，我这边还没补齐`

### 6.8 语气配比建议

为了保证产品既有性格，又保持可信度，建议控制为：

- 70% 清晰专业
- 20% 轻松自然
- 10% 有记忆点的俏皮

这样既能保留数据产品的可信度，也能让产品在球迷语境里更有辨识度。

## 7. 首页设计规范

### 7.1 首页定位

首页是 V1 的默认入口页，也是整个产品最重要的导航型首页。

首页不承担资讯门户职责，也不承担内容堆叠职责，而是负责：

- 承接首次访问
- 承接回访用户
- 提供搜索入口
- 分发到排名、赛事和球员浏览链路

首页的理想感受应是：

- 一眼就知道这是一个乒乓数据产品
- 一上来就能开始找信息
- 看起来轻盈、舒服，但不是空
- 有一点人格感，而不是冷启动工具页

### 7.2 首页目标

首页在产品层面需要完成以下目标：

- 让用户快速进入搜索链路
- 让用户快速浏览当前重点赛事
- 让用户快速进入排名浏览
- 让用户形成“这里比官方更顺手”的第一印象

首页在体验层面需要同时做到：

- 首屏有焦点
- 信息不拥挤
- 模块节奏清晰
- 视觉轻盈但不单薄

### 7.3 首页信息优先级

首页只保留三个核心模块：

1. 搜索入口
2. 赛事日程
3. 世界排名 Top 20

优先级原则：

- 搜索入口是首页第一交互焦点
- 赛事日程是首页第一视觉内容模块
- 排名 Top 20 是首页第一数据浏览模块

不建议在首页加入过多扩展信息，例如：

- 新闻流
- 多个次级榜单
- 冗长介绍文案
- 复杂运营位

### 7.4 首页整体结构

首页建议采用以下结构顺序：

1. 顶部头部摘要区
2. 搜索入口
3. 赛事日程模块
4. 排名 Top 20 模块
5. 底部导航安全空间

结构原则：

- 首屏必须同时看见头部和搜索入口
- 搜索与赛事之间应形成自然视觉衔接
- 排名模块应作为向下滚动后的主要内容承接

### 7.5 首页头部摘要区

首页头部的任务不是做传统 hero banner，而是建立产品气质与页面焦点。

应承担：

- 产品名称或产品定位
- 一句简短、自然、有点性格的引导文案
- 与搜索框的视觉承接

头部应给用户的感觉是：

- 这里是懂球的数据入口
- 可以查，也可以逛
- 不是硬邦邦的榜单页

内容建议：

- 保留一句轻量产品说明
- 避免长段介绍
- 避免泛化英文欢迎语

文案方向应更像陪看伙伴，而不是传统 app hero。

### 7.6 搜索入口设计

搜索入口是首页最重要的主交互组件。

#### 目标

- 成为首屏第一操作点
- 承接自然语言问题输入
- 在未登录状态下完成登录引导
- 在视觉上形成首页记忆点

#### 结构建议

- 搜索图标
- 输入提示文本
- 示例问题或提示文案
- 登录状态提示或轻功能提示

#### 交互规则

- 未登录用户可见但不可执行
- 未登录点击搜索框后直接进入登录页
- 已登录用户可输入并提交问题
- 提交后进入搜索结果页或结构化结果页

#### 文案建议

搜索框内外文案应避免冷冰冰的工具风。

例如可采用：

- `想查谁，直接问`
- `你问，我去翻数据`
- `试试：孙颖莎、王曼昱最近 3 年交手记录`

#### 视觉建议

- 可采用轻微磨玻璃和微透明悬浮质感
- 需保证边界、对比度和点击感清晰
- 不应做成过度发白、边界模糊的漂浮层

### 7.7 赛事日程模块

赛事日程模块是首页第一内容模块，应优先于排名列表出现。

#### 模块目标

- 承接用户对近期赛事的浏览需求
- 增强首页的“活内容感”
- 作为赛事页的自然入口

#### 展示方式

- 以月历卡片方式展示重点赛事
- 一个月一张月卡
- 支持放大查看和月份切换

#### 模块结构

- 模块标题
- 简短说明
- 月卡横向滑动区
- 放大查看交互

#### 交互规则

- 点击月卡进入放大月历态
- 放大态支持左右切换月份
- 点击浮层外区域关闭
- 点击放大态内容进入赛事列表页

#### 视觉要求

- 月卡应具备明显节奏感和浏览欲望
- 当前月或重点月需有更高识别度
- 放大态应有更强层级和沉浸感

该模块可以比数据模块更有一点画面感，但仍应保持结构清晰。

### 7.8 排名 Top 20 模块

排名模块是首页最重要的数据承接模块。

#### 模块目标

- 让用户快速浏览当前世界排名
- 提供球员详情页入口
- 提供进入完整排名页的主跳转

#### 展示字段

- 排名
- 头像
- 中文名
- 国家
- 积分
- 排名变化

说明：

- 首页 Top 20 列表默认不展示英文名，优先保证移动端信息密度和首屏节奏
- 英文名展示放在排名页等完整榜单视图中承接

#### 结构建议

- 模块标题
- 排名说明或更新时间提示
- Top 20 列表
- `查看更多` 入口

#### 交互规则

- 点击球员进入球员详情页
- 点击 `查看更多` 进入排名页
- 首页排名列表不承担双球员勾选功能

#### 视觉要求

- 信息密度高，但必须保持顺滑浏览感
- 球员身份信息和排名优先被看见
- 排名变化是辅助信息，不与主数值抢权重

### 7.9 首页模块节奏

首页必须避免“模块只是顺序堆叠”的感觉，而应形成明确节奏：

- 头部建立气质
- 搜索建立动作焦点
- 赛事建立浏览兴趣
- 排名建立数据价值

节奏要求：

- 首屏不拥挤
- 第二屏开始进入数据浏览状态
- 模块之间有明确留白与切换感

首页的观感应体现：

- 有呼吸感
- 有轻重层次
- 有内容吸引力

### 7.10 首页人格化表达建议

首页是最适合体现产品人格的页面之一，但表达应轻量，不应喧宾夺主。

适合体现人格的区域：

- 头部引导文案
- 搜索提示语
- 空状态提示
- 赛事模块辅助说明

不建议人格化过重的区域：

- 排名数据字段
- 比分、积分等核心数据

首页文案应给用户的感觉是：

- 有人在陪你看球、查球
- 但不会抢戏
- 有点好玩，但不轻浮

### 7.11 首页空状态与异常状态

首页应覆盖以下状态：

- 未登录搜索限制状态
- 赛事数据为空
- 排名数据为空
- 模块加载中
- 模块加载失败

规则：

- 空状态提供明确反馈
- 尽量给出下一步动作
- 不因数据未到而破坏首页结构

例如：

- 赛事为空时，仍保留模块框架与说明
- 排名为空时，保留模块标题与反馈入口

### 7.12 首页实现注意事项

首页实现时应特别注意：

- 搜索框、浮动导航、月历放大态之间的层级关系
- 移动端首屏高度控制
- 模块之间的加载和滚动节奏
- 顶部浅蓝氛围与正文数据区之间的视觉过渡
- 所有颜色和字体引用主题变量

首页是产品气质的第一次落地页面，不能只“能用”，还必须让用户愿意继续往下逛。
