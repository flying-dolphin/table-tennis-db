# ITTF 数据查询产品 PRD（V1）

版本：v1.0
日期：2026-04-17

---

## 1. 产品定位与目标

### 1.1 产品定位
本产品是一个面向乒乓球球迷的数据查询工具，聚焦 ITTF/WTT 相关赛事、球员、排名和比赛记录的数据整理、查询与展示。

产品不是替代官方数据源，而是在保留官方事实基础上，提供更适合球迷使用的数据浏览体验、更清晰的中文信息、更友好的交互方式，以及更符合球迷需求的统计维度和查询能力。

### 1.2 要解决的问题
当前官方 ITTF/WTT 网站主要存在以下问题：

- 用户体验较差，页面结构和交互方式老旧
- 缺少完整、统一的中文信息展示
- 球迷关心的数据维度支持不足，尤其是比赛记录、交手统计、个性化指标
- 缺少更灵活的查询能力

本产品的核心目标是：
为球迷提供一个更方便查询乒乓球数据的入口，让用户能够更轻松地查看球员、赛事、排名、比赛记录和交手数据，并逐步支持球迷真正关心的统计维度与高级查询能力。

### 1.3 目标用户
V1 阶段的核心用户包括：

- 普通球迷：查看排名、赛事、球员基础信息
- 深度球迷：关注比赛记录、交手历史、外战情况、对手分布等
- 数据型用户：希望基于现有数据做比较、统计和进一步查询

### 1.4 第一优先用户
V1 的第一优先用户是数据型用户。

原因：

- 对官方产品不满最明确
- 对比赛记录、交手统计、对比分析需求最强
- 更容易判断产品是否真正有价值
- 后续自然语言搜索和潜在付费能力更可能由这类用户驱动

### 1.5 产品目标
V1 的目标包括：

- 建立一个比官方站点更易用的中文数据查询入口
- 让用户能够方便查看比赛记录、交手统计、球员信息和赛事信息
- 验证用户是否愿意把本产品作为日常数据查询入口
- 验证用户对更高级查询能力的真实需求

### 1.6 差异化价值
相较官方站点，本产品的差异化价值主要体现在：

- 更好的中文信息支持
- 更友好的界面和交互体验
- 更方便查看比赛记录和交手统计
- 更贴近球迷需求的数据维度
- 支持更灵活的自然语言查询入口

### 1.7 成功标准
V1 是否成功，重点看以下方向：

- 有真实用户愿意持续使用
- 用户认为本产品在数据查询体验上明显优于官方站点
- 比赛记录、交手统计等核心功能成为主要使用理由
- 用户对高级查询能力表现出明确需求

说明：

- V1 不以成熟商业化功能上线为目标
- 当前规划中的高级付费能力主要是自然语言查询的增强版本
- 该能力预计在后续版本继续探索

## 2. V1 范围与非目标

### 2.1 V1 产品范围
V1 聚焦“女子单打数据查询工具”的核心闭环，并提供一个最简版自然语言搜索能力。

V1 覆盖：

- 首页
- 排名页
- 球员详情页
- 赛事列表页
- 赛事详情页
- 比赛详情页
- 搜索结果页
- 登录/注册页

### 2.2 V1 必做页面
#### 首页
包含：

- 搜索框
- 赛事日程
- 世界排名 Top 20

#### 排名页
必须支持：

- 按积分排序
- 按胜率排序
- 按对手交手次数排序
- 勾选两名球员进入对比

#### 球员详情页
必须包含：

- 基本信息
- 头像
- 当前排名 / 积分
- 核心统计
- 最近比赛
- 比赛记录
- 常见对手 / 交手统计

#### 赛事列表页
用于按年份浏览赛事。

#### 赛事详情页
用于展示某个赛事下不同 `sub event` 的结果结构，并以对战图为主要呈现方式。

#### 比赛详情页
用于展示单场比赛的详细信息。

#### 搜索结果页
用于承接自然语言搜索结果。

#### 登录/注册页
用于注册、登录、登出和当前用户信息承接。

### 2.3 搜索与账号范围
#### 搜索功能
V1 提供最小可用的自然语言搜索：

- 仅支持少量高价值问题类型
- 不做效果调优
- 已登录用户可用
- 未登录用户可见入口但不可执行

#### 账号功能
V1 支持：

- 注册
- 登录
- 记住登录状态
- 登出
- 查看当前用户信息

### 2.4 V1 非目标
V1 明确不做：

- 收藏
- 评论 / 社区互动
- 数据导出
- 多项目种全面覆盖
- 实时比分 / 实时推送
- 自定义看板
- 搜索效果复杂调优
- 完整付费体系
- 修改密码
- 找回密码
- 第三方登录

### 2.5 V1 与后续版本边界
V1 重点是验证核心数据查询需求和搜索需求。

后续版本可继续扩展：

- 更完善的自然语言搜索
- 搜索效果优化
- 更多项目种
- 世界冠军总数等新增指标
- 更完整账号体系

## 3. 用户与使用场景

### 3.1 用户分层判断
上线前暂不预设单一用户群体，V1 同时服务：

- 普通球迷
- 深度球迷
- 数据型用户

### 3.2 核心使用场景
典型场景包括：

- 看比赛期间，临时查询两名球员的历史交手记录
- 比较两个球员近几年面对某类对手时的表现差异
- 比较两个球员在某类统计指标上的差别
- 查询某个球员某场比赛的结果
- 浏览某个球员近期比赛和交手对象
- 浏览赛事对战图和晋级路径

### 3.3 主要访问入口
V1 的主要访问入口为首页。

首页承担：

- 首次访问承接页
- 回访用户默认入口
- 通往排名、赛事、搜索的导航枢纽

### 3.4 用户最想完成的任务
V1 中用户最核心的任务包括：

- 对比两个球员的数据
- 查看两个球员的历史交手记录
- 查询某个球员某场比赛的结果

### 3.5 未登录用户的可访问范围
未登录用户可浏览：

- 首页
- 排名页
- 球员页
- 赛事页
- 赛事详情页
- 比赛详情页

未登录用户不可执行搜索。

### 3.6 用户验证目标
V1 需要验证：

- 数据型用户是否会成为最稳定用户群体
- 普通球迷是否愿意把本产品作为官方站点替代入口
- “内容开放浏览 + 登录解锁搜索”的模式是否成立
- 哪些查询场景最能驱动用户反复使用

## 4. 信息架构与页面地图

### 4.1 整体信息架构原则
V1 采用移动端优先的信息架构，交互风格更接近移动应用。

核心原则：

- 首页承担主入口角色
- 球员、赛事、比赛之间形成自然跳转闭环
- 搜索是高级能力入口，不是唯一入口

### 4.2 全局导航结构
采用底部导航栏，固定 4 个入口：

- 首页
- 排名
- 赛事
- 用户入口

不设置“球员”独立导航入口。

球员页主要通过以下路径进入：

- 排名页
- 赛事页 / 赛事详情页
- 搜索结果页

### 4.3 首页定位
首页是导航型首页，仅保留三个核心模块：

- 搜索框
- 赛事日程
- 运动员排名

### 4.4 页面地图
核心页面包括：

- `/`
- `/rankings`
- `/players/[slug]`
- `/compare`
- `/events`
- `/events/[eventId]`
- `/matches/[matchId]`
- `/search`
- `/auth`

### 4.5 核心页面关系
主要关系如下：

- 首页 -> 排名页 -> 球员详情页
- 首页 -> 排名页 -> 勾选两个球员 -> 对比页
- 首页 -> 赛事列表页 -> 赛事详情页 -> 比赛详情页
- 首页 -> 搜索结果页 -> 球员页 / 赛事页 / 比赛页 / 对比页
- 球员详情页 -> 比赛详情页
- 赛事详情页 -> 比赛详情页

### 4.6 核心主链路
V1 最重要的主链路：

首页 -> 排名页 -> 选择两个球员 -> 对比视图

### 4.7 球员对比入口设计
V1 的结构化双球员对比主入口放在排名页：

- 勾选两名球员
- 激活对比动作
- 跳转到独立对比页

### 4.8 搜索承接方式
搜索结果通过独立搜索结果页承接。

规则：

- 如果问题可完全映射到结构化对比页，则直接跳转对比页
- 如果是混合问题，则搜索结果页先展示结构化结果，再展示搜索结果表格
- 如果无法映射结构化结果，则显示搜索结果页

### 4.9 赛事详情与比赛详情边界
- 赛事详情页：展示某个 `event` 下各 `sub event` 的正赛对战图和冠军
- 比赛详情页：展示某个 `match` 的单场详细信息

### 4.10 未登录用户在信息架构中的位置
内容浏览链路默认开放，搜索链路登录后开放。

## 5. 首页与全局框架需求

### 5.1 全局框架定位
V1 采用移动端优先的应用式框架，包含：

- 顶部页面头部区域
- 底部导航栏
- 全局登录态控制
- 搜索入口
- 统一跳转规则

### 5.2 底部导航栏
#### 未登录状态
最后一个入口显示头像占位，点击进入登录页。

#### 已登录状态
最后一个入口进入用户页，仅承载：

- 用户名
- 邮箱
- 注册时间
- 退出登录

### 5.3 首页目标
首页目标：

- 作为主要入口
- 快速承接搜索、赛事浏览、排名浏览
- 保持轻量，不做内容堆叠

### 5.4 首页搜索框需求
首页默认展示搜索框和示例问题。

#### 未登录状态
- 搜索框可见但不可直接使用
- 显示示例问题
- 点击后直接打开登录页

#### 已登录状态
- 可输入问题
- 提交后进入 `/search`

### 5.5 首页赛事日程模块
首页赛事日程按月展示为日历图。

规则：

- 一个月一张月卡
- 展示该月重点赛事
- 后端按赛事分类过滤重点赛事
- 前端根据赛事列表组装成月历图
- 点击月卡：进入放大日历图查看态
- 放大态左右滑动：切换月份
- 点击放大的日历图：关闭

### 5.6 首页运动员排名模块
首页展示 Top 20，字段包括：

- 排名
- 头像
- 中文名
- 英文名
- 国家
- 积分
- 排名变化

提供“查看更多”进入排名页。

### 5.7 全局登录限制规则
- 内容页公开
- 搜索功能登录后可用
- 未登录点击搜索框直接进入登录页

### 5.8 全局跳转原则
- 首页排名模块球员 -> 球员详情页
- 首页查看更多 -> 排名页
- 首页赛事日程 -> 放大日历图 
- 搜索框 -> 登录页或搜索结果页

## 6. 排名页与双球员对比需求

### 6.1 页面定位
排名页既承担基础排名浏览能力，也承担双球员结构化对比主入口。

### 6.2 页面目标
- 查看最新女子单打排名
- 按不同统计维度重新排序
- 进入球员详情页
- 发起双球员对比

### 6.3 数据范围
默认展示：

- 女子单打
- 最新一期排名

V1 不做复杂筛选。

### 6.4 排序能力
必须支持：

- 按积分排序
- 按胜率排序
- 按对手交手次数排序

说明：

- 默认按积分
- 后两者按全量历史统计
- 切换排序时仍返回原始排名和积分

### 6.5 页面结构
包含：

- 页面标题区
- 排序切换区
- 排名表
- 双球员勾选与对比操作区

### 6.6 排名表要求
排名表需支持：

- 展示最新女子单打排名
- 点击球员进入详情页
- 勾选球员用于对比
- 返回纯数据

字段至少包括：

- 排名
- 球员姓名
- 头像
- 国家
- 积分
- 排名变化值
- 当前排序字段值
- 跳转 profile 所需标识

### 6.7 双球员勾选规则
- 只支持两人对比
- 勾选 2 人后激活对比
- 不支持 3 人及以上同时对比

### 6.8 对比页定位
对比页是 V1 数据型用户最重要的结果页之一。

### 6.9 对比页最小必备内容
必须展示：

- 排名
- 积分
- 胜率
- 外战胜率
- 内战胜率
- 三大赛冠军总数
- 七大赛冠军总数
- `events` 总数
- 七大赛进决赛次数
- 历史交手记录
- 历史交手胜率

V1 不展示世界冠军总数。

### 6.10 对比指标说明
所有对比指标必须使用统一口径，见第 10 章。

### 6.11 对比页交互要求
- 支持查看结构化指标对比
- 支持查看历史交手记录
- 支持更换其中一个球员
- 支持点击球员进入详情页
- 支持点击历史交手比赛进入比赛详情页

### 6.12 页面跳转关系
- 排名页 -> 球员详情页
- 排名页 -> 对比页
- 对比页 -> 球员详情页
- 对比页 -> 比赛详情页

### 6.13 V1 不做的内容
- 时间范围筛选
- 国家筛选
- 多人对比
- 自定义指标配置
- 导出

### 6.14 验收标准
- 三种排序可用
- 可勾选两名球员进入对比页
- 对比页完整展示核心指标
- 支持更换其中一人
- 历史交手比赛可跳详情页

## 7. 球员详情页需求

### 7.1 页面定位
球员详情页用于承接排名页、赛事页、比赛页和搜索结果进入后的单球员深入查看需求。

V1 只面向女子单打球员。

### 7.2 页面目标
- 展示球员核心信息
- 展示当前排名与积分
- 展示统一口径的核心统计
- 展示最近比赛
- 展示赛事记录
- 展示 Top 3 对手

### 7.3 页面顶部信息区
顶部展示：

- 头像
- 中文名
- 英文名
- 国家
- 当前排名
- 当前积分

### 7.4 基础信息与核心统计区
统计指标与对比页保持一致，包括：

- 排名
- 积分
- 胜率
- 外战胜率
- 内战胜率
- 三大赛冠军总数
- 七大赛冠军总数
- `events` 总数
- 七大赛进决赛次数

### 7.5 最近比赛模块
最近比赛定义为最近 3 场 `match`。

每场展示：

- 所属 `event name`
- 时间
- 对手
- 最终比分
- 是否获胜

### 7.6 比赛记录模块
比赛记录在 V1 中按时间倒序展示 `events list`，而非完整逐场 match 列表。

每条至少包括：

- 赛事名称
- 时间
- 赛事项目
- 成绩 / 最终轮次
- 详情跳转标识

V1 不支持筛选。

### 7.7 交手统计补充模块
展示交手次数最多的对手 Top 3。

每条至少包括：

- 交手次数
- 胜率
- 最近一次交手时间
- 对手详情跳转

### 7.8 页面交互要求
- 最近比赛 -> 比赛详情页
- 赛事记录 -> 赛事详情页
- Top 3 对手 -> 对手详情页

V1 不支持在球员详情页直接发起双球员对比。

### 7.9 页面边界
- 只面向女子单打
- 不支持复杂筛选
- 不做高级分析工具

### 7.10 数据一致性要求
同名指标需与：

- 排名页
- 对比页
- 搜索结果中的结构化结果

保持一致。

### 7.11 验收标准
- 顶部信息正确显示
- 核心统计完整
- 最近比赛固定显示最近 3 场
- 比赛记录按时间倒序展示
- Top 3 对手展示正确
- 跳转关系正常

## 8. 赛事列表页、赛事详情页与比赛详情页需求

### 8.1 页面定位
这一组页面承担按赛事维度浏览数据的主链路。

概念必须严格区分：

- `event`：赛事
- `sub event`：赛事项目
- `match`：具体比赛

### 8.2 赛事列表页
#### 目标
按年份浏览历史赛事。

#### 浏览方式
- 按年份切换
- 当前支持回溯到 2014 年
- 不做复杂筛选

#### 作用
- 提供完整历史赛事浏览入口
- 作为赛事详情页入口

### 8.3 赛事详情页
#### 目标
展示某个 `event` 下不同 `sub event` 的结果结构和冠军信息。

#### 顶部基础信息
只展示：

- 赛事名称
- 时间

#### `sub event` 支持
允许展示多个 `sub event`，如：

- 女单
- 女双
- 混双
- 女团

需要提供 `sub event` 入口，并展示各 `sub event` 的最终冠军。

#### 默认展示规则
- 普通赛事默认女单
- 团体赛默认女子团体

#### 核心展示内容
对于当前选中的 `sub event`，展示：

- 最终冠军
- 正赛对战图
- 晋级路径

#### 交互规则
点击对战图中的任意比赛，进入比赛详情页。

### 8.4 比赛详情页
#### 目标
展示某个 `match` 的单场详细信息。

#### 最小必备字段
- 赛事名称
- `sub event`
- 轮次 / 阶段
- 比赛日期
- 双方球员
- 国家
- 最终比分
- 局分明细
- 获胜方
- 返回所属赛事入口

#### 页面边界
不展示前一轮 / 下一轮等复杂上下文。

### 8.5 页面跳转关系
- 赛事列表页 -> 赛事详情页
- 赛事详情页切换 `sub event`
- 赛事详情页对战图 -> 比赛详情页
- 比赛详情页 -> 所属赛事入口

### 8.6 多 `sub event` 支持边界
- 赛事详情页允许多个 `sub event`
- 比赛详情页跟随所属 `sub event` 展示
- 球员详情页仍以女子单打球员为主

### 8.7 验收标准
- 赛事列表页可按年份切换
- 赛事详情页支持 `sub event` 切换
- 默认项目规则正确
- 正赛对战图可用
- 比赛详情页字段完整
- 跳转正确

## 9. 搜索、登录与访问控制需求

### 9.1 模块定位
V1 提供最简版自然语言搜索，并通过登录控制其访问权限。

### 9.2 搜索功能定位
搜索是受限开放的高级功能，特点：

- 支持简单自然语言查询
- 仅登录用户可用
- 不做复杂调优
- 覆盖少量高价值问题类型

### 9.3 搜索入口规则
#### 未登录状态
- 可看到搜索框和示例问题
- 搜索框不可直接使用
- 点击后直接打开登录页

#### 已登录状态
- 可输入问题
- 进入独立搜索结果页或直接跳转结构化对比页

### 9.4 V1 支持的问题类型
优先支持：

- 两名球员最近 3 年交手记录
- 两名球员分别与第三名球员的交手次数、胜率对比
- 两名球员最近 3 年和某国家对手的交手记录对比
- 两名球员外战胜率对比

### 9.5 搜索结果页形态
#### 规则
- 顶部显示用户原问题
- 如果只有结构化结果，则直接跳结构化页
- 如果是混合结果，则搜索页分两部分：
  - 结构化结果
  - 搜索结果表格

#### 搜索结果表格
用于承接非结构化部分结果。

### 9.6 不支持问题的处理方式
对于 V1 不支持的问题类型，明确提示：

`暂不支持该问题类型`

### 9.7 登录与注册能力
#### 注册
- 同一页面内切换登录 / 注册
- 注册流程：
  - 输入邮箱、用户名
  - 发送验证码
  - 设置密码
  - 同页完成注册

#### 登录
- 邮箱 + 密码

#### 登录态
- 记住登录状态
- 30 天失效

### 9.8 访问控制规则
- 内容页默认公开
- 搜索接口必须登录
- 未登录访问搜索接口返回未授权
- 登录后回首页

### 9.9 用户入口最小范围
用户入口只展示：

- 用户名
- 邮箱
- 注册时间

并提供退出登录。

### 9.10 错误提示规则
以下错误返回具体原因：

- 验证码校验错误
- 用户名重复
- 密码不一致
- 邮箱已被注册

其他错误：

- 前端显示通用错误提示
- 后台记录实际日志

### 9.11 验收标准
- 注册可用
- 登录可用
- 记住登录状态 30 天
- 搜索权限控制正确
- 支持的问题类型有结果
- 不支持的问题类型有明确提示

## 10. 全局数据口径与统计指标定义

### 10.1 章节目标
统一所有页面和接口中的关键统计口径。

### 10.2 适用范围
V1 默认适用于：

- 女子单打球员
- 当前已入库可用数据
- 默认全量历史数据，除非问题明确指定时间范围

### 10.3 排名口径
#### 当前排名
女子单打最新一期排名名次。

#### 当前积分
女子单打最新一期排名积分。

### 10.4 比赛与赛事口径
必须明确区分：

- `event`
- `sub event`
- `match`

### 10.5 胜率口径
#### 胜率
全量历史 `match` 胜率。

计算：

- 分子：获胜场次
- 分母：有明确胜负结果的总场次

### 10.6 外战与内战口径
#### 外战胜率
对不同国家对手的历史 `match` 胜率。

#### 内战胜率
对同国家对手的历史 `match` 胜率。

### 10.7 交手记录口径
#### 历史交手记录
两名球员之间所有历史 `match` 记录。

#### 历史交手胜率
两名球员互相交手时各自的胜率。

### 10.8 交手次数最多对手口径
指某球员历史范围内交手场次最多的 3 位对手。

建议排序：

1. 交手场次降序
2. 最近交手时间更近优先

### 10.9 赛事参与次数口径
#### `events` 总数
球员参与过的赛事总数，按 `event` 去重统计。

### 10.10 三大赛 / 七大赛口径
基于 `event_categories.sort_order`：

- 三大赛：`sort_order` 在 `1-3`
- 七大赛：`sort_order` 在 `1-7`

#### 三大赛冠军总数
按三大赛口径统计冠军数。

#### 七大赛冠军总数
按七大赛口径统计冠军数。

#### 七大赛进决赛次数
按七大赛口径统计进入决赛次数，包含冠军。

#### 世界冠军总数
V1 暂不计算，后续版本预留。

### 10.11 搜索与结构化对比结果路由规则
1. 判断问题是否存在结构化对比指标
2. 若已有结构化对比页支持，则优先跳转对比页
3. 若不支持，则进入搜索结果页
4. 若结构化与非结构化混合，则输出混合结果

### 10.12 缺失数据处理规则
- 显示“暂无数据”
- 不用 `0` 伪装缺失值
- 如可行，提示统计基于当前已收录数据范围

### 10.13 一致性要求
以下页面中的同名指标必须口径一致：

- 排名页
- 对比页
- 球员详情页
- 搜索结果页

## 11. 后端接口需求

### 11.1 章节目标
定义前后端接口边界，支持页面渲染和搜索能力。

### 11.2 统一接口设计原则
- REST 风格
- 路径统一使用 `/api/v1/...`
- 列表页只返回列表所需字段
- 详情页优先单接口返回整页所需数据
- 搜索由统一接口处理
- 公开内容接口无需登录
- 搜索接口需登录

### 11.3 统一响应格式
#### 成功
```json
{
  "code": 0,
  "message": "ok",
  "data": {},
  "meta": {}
}
```

#### 失败
```json
{
  "code": 1001,
  "message": "error message",
  "error": {
    "type": "validation_error",
    "detail": "..."
  }
}
```

### 11.4 权限规则
- 内容接口公开
- 搜索接口需登录
- 注册 / 登录 / 登出 / me 为账号相关接口

### 11.5 首页摘要接口
首页采用两个可复用接口：

#### 赛事日程接口
返回重点赛事列表，由前端组装成月历图。

#### Top 20 排名接口
返回首页 Top 20 摘要排名数据。

### 11.6 排名页接口
`GET /api/v1/rankings`

参数建议：

- `category=women_singles`
- `sort_by=points|win_rate|head_to_head_count`

返回列表所需字段：

- `player_id`
- `slug`
- `name`
- `name_zh`
- `avatar`
- `country_code`
- `rank`
- `rank_change`
- `points`
- 当前排序字段值

### 11.7 双球员对比接口
`GET /api/v1/compare`

参数：

- `player_a`
- `player_b`

返回：

- 两名球员基础信息
- 核心统计
- 历史交手记录
- 历史交手汇总统计

历史交手记录每条至少包括：

- 日期
- 名称
- 轮次
- 比分
- 详情跳转标识

### 11.8 球员详情接口
`GET /api/v1/players/{slug}`

单接口返回整页所需数据，包括：

- 顶部基础信息
- 核心统计
- 最近比赛
- 比赛记录
- Top 3 对手

### 11.9 赛事列表接口
`GET /api/v1/events?year=...`

返回年份下赛事列表，支持回溯到 2014。

### 11.10 赛事详情接口
`GET /api/v1/events/{eventId}`

支持参数：

- `sub_event`

返回：

- 赛事名称
- 时间
- `sub event` 列表
- 各 `sub event` 冠军
- 当前 `sub event` 对战图和晋级路径

### 11.11 比赛详情接口
`GET /api/v1/matches/{matchId}`

返回：

- 赛事名称
- `sub event`
- 轮次 / 阶段
- 日期
- 双方球员
- 国家
- 最终比分
- 局分明细
- 获胜方
- 所属赛事入口信息

### 11.12 搜索接口
`POST /api/v1/search`

请求：

- `query`

后端负责判断：

- `redirect_compare`
- `search_result`
- `mixed_result`
- `unsupported`

### 11.13 注册、登录与用户接口
#### 注册
`POST /api/v1/auth/register`

字段：

- 邮箱
- 用户名
- 验证码
- 密码
- 确认密码

#### 登录
`POST /api/v1/auth/login`

字段：

- 邮箱
- 密码

#### 登出
`POST /api/v1/auth/logout`

#### 当前用户
`GET /api/v1/auth/me`

### 11.14 反馈日志接口
统一接口，上报当前页面接口请求参数。

V1 不做去重和频控。

### 11.15 接口与页面映射关系
- 首页：赛事日程接口 + Top20 排名接口
- 排名页：`/rankings`
- 对比页：`/compare`
- 球员详情页：`/players/{slug}`
- 赛事列表页：`/events?year=...`
- 赛事详情页：`/events/{eventId}`
- 比赛详情页：`/matches/{matchId}`
- 搜索结果页：`/search`
- 账号：`/auth/...`

### 11.16 验收标准
- 所有页面有稳定接口支撑
- 列表页只返回列表字段
- 详情页单接口可支撑整页
- 搜索统一接口可路由不同结果
- 注册/登录接口可用
- 搜索接口权限正确

## 12. 全局交互规则、空状态与异常处理

### 12.1 全局状态原则
- 不隐瞒数据缺失
- 不用 `0` 伪装真实数据
- 尽量提供反馈入口

### 12.2 数据缺失通用处理规则
缺失时显示：

- `暂无数据`
- `点击上报`

### 12.3 用户反馈与上报入口
统一为一个按钮。

行为：

- 前端发送当前页面接口请求参数到服务端
- 成功后提示“已反馈”
- 提示悬浮几秒自动消失

### 12.4 部分 `sub event` 无数据处理规则
- 所有 `sub event` 入口仍显示
- 无数据入口置灰

### 12.5 最近比赛不足处理规则
如果最近比赛数据不足：

- 模块显示 `暂无数据`
- 提供 `想要` 入口

### 12.6 搜索无结果处理规则
显示：

- `没有找到结果`
- `点击反馈`

### 12.7 不支持问题类型处理规则
显示：

- `暂不支持该问题类型`

### 12.8 页面加载状态
V1 采用简单 loading，不强制 skeleton。

### 12.9 登录受限交互规则
未登录点击搜索框：

- 直接跳登录页

### 12.10 验收标准
- 缺失数据显示 `暂无数据`
- 搜索无结果显示 `没有找到结果，点击反馈`
- 不支持问题类型有明确提示
- 无数据 `sub event` 置灰
- 最近比赛缺失有 `想要` 入口
- 页面加载有简单 loading

- 登录接口限流
- 搜索接口限流

### 13.6 上线判断原则

- 数据口径一致
- 登录与搜索边界清晰
